# 代码漏掉事件复盘报告

## 事件概述

在开发红利类指数打分功能时，由于操作失误导致 `cli/lib/api.js` 文件中多个重要方法丢失，包括：
- `getMarketData`
- `getMarketTemp`
- `getMarketStyle`
- `getMarketValueData`
- `getBkData`
- `getSectorData`
- `getSectorRankStock`
- `getTopStocks`
- `getGnHot`
- `getStockData`
- `getGainianStock`
- `getKline`
- `getZdtPool`
- `getSecId`
- `queryStockData`
- `getPatternStocks`

## 问题根因分析

### 1. 代码修改方式不当
**问题描述**：在添加新功能时，直接重写了整个文件，而不是在原有基础上进行增量修改。

**具体表现**：
- 使用 `Write` 工具完全覆盖了原有文件
- 没有先读取现有文件内容
- 没有保留原有的函数和方法

**影响范围**：导致所有原有的API方法丢失，影响了多个CLI命令的正常使用。

### 2. 缺少代码审查环节
**问题描述**：在修改代码后，没有进行充分的代码审查和测试。

**具体表现**：
- 修改后只测试了新增功能
- 没有运行完整的测试套件
- 没有检查其他依赖该文件的模块

**影响范围**：问题在用户使用时才被发现，增加了修复成本。

### 3. 对DRY原则理解不足
**问题描述**：在重构代码时，没有遵循DRY（Don't Repeat Yourself）原则。

**具体表现**：
- 每个API方法都重复编写了axios请求代码
- 没有使用 `createClient` 和 `post`/`get` 辅助函数
- 代码冗余度高，维护困难

**影响范围**：增加了代码维护成本，容易引入错误。

### 4. Git操作不熟练
**问题描述**：在处理合并冲突时，没有正确保留所有代码。

**具体表现**：
- 没有仔细检查合并后的文件内容
- 没有验证所有方法是否都存在
- 没有运行测试验证功能完整性

**影响范围**：导致部分功能丢失。

## 暴露的问题

### 1. 代码修改流程问题
- ❌ 缺少"先读后改"的习惯
- ❌ 缺少代码审查环节
- ❌ 缺少完整的测试验证

### 2. 代码质量意识问题
- ❌ 对DRY原则理解不足
- ❌ 对代码副作用认识不足
- ❌ 对错误处理重视不够

### 3. 工具使用问题
- ❌ 对Git操作不熟练
- ❌ 对文件编辑工具使用不当
- ❌ 对代码验证工具使用不足

## 改进措施

### 1. 代码修改流程改进

#### 1.1 强制执行"先读后改"原则
```
修改任何现有文件前，必须：
1. 使用 Read 工具读取文件完整内容
2. 理解现有代码结构和逻辑
3. 确认修改的影响范围
4. 使用 Edit 工具进行增量修改（而不是 Write）
```

#### 1.2 增加代码审查环节
```
每次代码修改后，必须：
1. 检查修改的完整性
2. 验证所有依赖项是否正常
3. 运行相关测试用例
4. 检查是否有未使用的变量或导入
```

#### 1.3 建立测试验证机制
```
每次功能修改后，必须：
1. 测试新增功能
2. 测试受影响的现有功能
3. 运行完整的测试套件（如果有）
4. 验证边界情况和异常处理
```

### 2. 代码质量标准

#### 2.1 遵循DRY原则
```
- 提取重复代码为公共函数
- 使用辅助函数减少代码冗余
- 保持代码简洁和可维护
```

#### 2.2 避免副作用
```
- 不直接修改传入的参数
- 使用深拷贝处理数据
- 保持函数的纯度
```

#### 2.3 完善错误处理
```
- 添加参数验证
- 添加数据验证
- 提供清晰的错误消息
```

### 3. 工具使用规范

#### 3.1 Git操作规范
```
- 合并前检查冲突内容
- 合并后验证文件完整性
- 使用 git diff 检查变更
```

#### 3.2 文件编辑规范
```
- 优先使用 Edit 工具
- 谨慎使用 Write 工具
- 修改前备份重要文件
```

## LLM行为准则更新

基于此次事件，需要在LLM行为准则中添加以下内容：

### 1. 代码修改行为准则

**强制要求**：
1. **先读后改**：修改任何现有文件前，必须先使用 Read 工具读取文件完整内容
2. **增量修改**：优先使用 Edit 工具进行增量修改，避免使用 Write 工具完全覆盖
3. **完整性验证**：修改后必须验证所有原有功能是否完整保留
4. **依赖检查**：检查修改是否影响其他依赖该文件的模块

### 2. 代码质量行为准则

**强制要求**：
1. **DRY原则**：提取重复代码为公共函数，避免代码冗余
2. **副作用控制**：不直接修改传入参数，使用深拷贝处理数据
3. **错误处理**：添加参数验证和数据验证，提供清晰的错误消息
4. **代码审查**：修改后进行代码审查，检查未使用的变量和导入

### 3. 测试验证行为准则

**强制要求**：
1. **功能测试**：测试新增功能和受影响的现有功能
2. **完整性测试**：运行完整的测试套件（如果有）
3. **边界测试**：验证边界情况和异常处理
4. **回归测试**：确保修改没有破坏现有功能

### 4. Git操作行为准则

**强制要求**：
1. **合并前检查**：仔细检查合并冲突内容
2. **合并后验证**：验证合并后的文件完整性
3. **变更检查**：使用 git diff 检查变更内容
4. **提交前验证**：确保所有功能正常工作后再提交

## 测试用例需求

为了防止类似问题再次发生，需要创建完整的CLI测试用例：

### 测试范围
1. 所有CLI命令和子命令
2. 所有参数组合（除了lmt参数）
3. 错误处理和边界情况
4. Token验证和权限检查

### 测试方法
1. 使用用户提供的Token进行测试
2. 验证每个接口的返回数据结构
3. 测试参数验证和错误处理
4. 测试边界情况和异常场景

## 总结

此次事件暴露了在代码修改流程、代码质量意识和工具使用方面的不足。通过建立明确的行为准则和测试机制，可以有效避免类似问题的再次发生。

**关键教训**：
1. 永远不要直接覆盖现有文件，必须先读取再修改
2. 修改后必须验证所有功能的完整性
3. 遵循DRY原则，保持代码简洁可维护
4. 建立完善的测试机制，确保代码质量
