# 个股消息面催化分析 Skill

围绕“舆情 + 公告 + 研报”三类信息源，先做结构化抓取，再做官方信源交叉验证，最后按影响方向、信源可靠度、时间衰减进行排序打分，输出可执行的消息面汇报。

## Overview（功能概述）

该 Skill 使用 `daxiapi-cli` 的 `news` 命令快速获取个股消息面信息，并结合标题关键词进行外部官方信源检索，识别真假与重要性，形成“结论先行 + 证据可追溯”的完整报告。

## When to Use（何时使用）

当用户需要以下能力时触发：

- 分析某只股票近期消息面是利好还是利空
- 汇总舆情、公告、研报并进行优先级排序
- 对新闻真伪做交叉验证（官方信源优先）
- 输出结构化消息面跟踪报告

**具体触发词**：个股舆情、公告解读、研报解读、消息面分析、新闻催化、利好利空、新闻评分

## When Not to Use（何时不使用）

以下场景不应使用本 Skill：

- 用户未提供股票代码且无法确认标的
- 非A股标的（美股、港股、基金、期货、加密资产）
- 仅要求技术指标教学（如“MACD是什么”）
- 仅要求财务深度分析（应优先路由到 `xiapi-financial-roe-analysis`）

## Process（流程主体）

### Step 0: 前期准备（Token 与标的确认）

**触发条件**：首次使用、Token 未配置、或返回认证错误

**跳过条件**：Token 已配置且有效，且标的代码已明确

**完成标准**：能成功获取 `news` 数据

**步骤 0.1：检查 Token**

```bash
npx daxiapi-cli@latest config get token
```

**步骤 0.2：如未配置，指导配置 Token**

```bash
npx daxiapi-cli@latest config set token YOUR_TOKEN_FROM_DAXIAPI
# 或
export DAXIAPI_TOKEN=YOUR_TOKEN_FROM_DAXIAPI
```

**步骤 0.3：确认标的代码**

```bash
npx daxiapi-cli@latest search <公司名> --type stock
```

---

### Step 1: Agent 循环（获取 + 验证 + 评分）

**目标**：完成三类信息收集、官方信源交叉验证、影响评分与排序

#### 1.1 获取三类新闻数据

```bash
# 舆情（内部自动将 code 转为 secid）
npx daxiapi-cli@latest news sentiment -c <code> -p 30

# 公告
npx daxiapi-cli@latest news notice -c <code> -p 30 -i 1

# 研报（建议默认近90天）
npx daxiapi-cli@latest news report -c <code> -p 30 -i 1 -b <beginDate> -e <endDate>
```

#### 1.2 提取关键词并做官方信源交叉验证

对每条新闻标题提取关键词（公司名、核心事件词、数字词），按下列优先级做检索验证：

1. **交易所/法定披露**：上交所、深交所、巨潮资讯
2. **公司官方**：公司官网公告、投资者关系公告
3. **监管/权威机构**：证监会、国家统计局、行业协会
4. **主流财经媒体**：作为补充，不作为最高级证据

验证结果分级：

- `A`：有法定披露或交易所文件可对应
- `B`：有公司官方公告或权威机构信息可对应
- `C`：仅媒体/自媒体来源，未找到官方对应证据
- `D`：标题夸张、事实不清或与官方信息冲突

#### 1.3 情绪方向识别（正向/负向）

对标题与摘要进行关键词打标；可多标签共存：

- 负面词示例：`dump`, `bear`, `crash`, `loss`, `sell`, `scam`, `fear`, `volatile`
- 正面词示例：`moon`, `bull`, `growth`, `profit`, `buy`, `upgrade`, `success`, `stable`

规则：

- 仅作“辅助信号”，不得单独作为结论依据
- 若关键词信号与官方披露冲突，以官方披露为准

#### 1.4 影响评分模型（0-100）

对每条事件按三维打分后加权：

- **方向分 Direction（-1/0/+1）**：负向=-1，中性=0，正向=+1
- **信源分 Source（0-100）**：A=100，B=75，C=45，D=20
- **时效分 Recency（0-100）**：
  - 0-1天=100
  - 2-3天=85
  - 4-7天=70
  - 8-14天=50
  - 15-30天=30
  - 30天以上=15

事件最终分（带方向）：

`eventScore = Direction * (0.45 * Source + 0.55 * Recency)`

股票综合消息分：

`stockNewsScore = clamp(50 + sum(eventScore_i), 0, 100)`

- `0-35`：明显偏空
- `36-45`：偏空
- `46-55`：中性
- `56-70`：偏多
- `71-100`：明显偏多

#### 1.5 排序与去重

- 相同事件多来源重复时，仅保留“最高信源等级 + 最新时间”版本
- 按 `|eventScore|` 从高到低排序（先看影响强度，再看方向）
- 至少保留前 5 条高影响事件（若数据不足则如实说明）

---

### Step 2: 汇总报告

**目标**：输出可执行的个股消息面汇报，覆盖结论、证据、风险

**执行步骤**：

1. 输出综合结论（先给方向和置信度）
2. 列出高影响事件清单（含信源与验证状态）
3. 给出观察清单（未来1-2周需要跟踪的触发条件）
4. 补充风险提示与免责声明

## Common Pitfalls（常见陷阱）

- 只看媒体标题，不做官方信源验证
- 用关键词情绪直接替代事实判断
- 把旧闻当成新催化，忽略时效衰减
- 多来源重复报道导致“虚假高热度”

## Key Principles（重要原则）

1. 官方信源优先于媒体转述
2. 时效性优先于历史噪音
3. 证据优先于观点
4. 结论必须可解释、可追溯

## Error Handling（错误处理）

- **Token 错误（401）**：先执行 `config get token`，再引导重配 token
- **数据为空**：缩短时间区间、检查代码、降低页码过滤
- **信源无法验证**：标记为 `待验证`，降低信源分，不直接下强结论
- **新闻冲突**：并列呈现冲突观点，以法定披露作为裁决依据
